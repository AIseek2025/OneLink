# Product And Platform

## 1. 产品定位

OneLink 的中文名为“一度相连”。核心承诺是：用户只需要认识一个 AI 好朋友，就能在安全、合规、可解释的前提下，更高效地连接平台内最合适的人。

当前边界：OneLink 只匹配平台内已经注册、授权、可被推荐的人，不做全网人肉搜索，不抓取未授权个人隐私数据。

## 2. 北极星

- 对用户：更快找到真正有帮助、愿意回应、适合建立关系的人。
- 对平台：有效连接率持续提升，举报率和误伤率可控。
- 对技术：长期理解、低成本上下文、高并发服务、可审计安全闭环。

## 3. AI 好朋友入口

用户进入 OneLink 的第一个关系对象是 Lumi。Lumi 不是万能问答助手，她的主使命是帮助用户认识自己、表达需求、补齐画像、发起连接，并在高风险场景中守住边界。

## 4. 核心用户链路

MVP 必须围绕以下闭环推进：

1. 注册 / 登录。
2. 与 Lumi 对话。
3. 问卷和自然追问补齐画像。
4. 记忆抽取、压缩、整合。
5. 画像投影与用户可见纠正。
6. 用户表达找人意图。
7. 安全审查、候选召回、排序、名片返回。
8. 关注 / 私信 / 举报 / 反馈。
9. 反馈反哺画像、匹配、安全和策略。

## 5. 三端产品策略

### 5.1 第一阶段：App + Web 双端同步

前期以 App 和 Web 同步开发为主：

- App 负责高频聊天、推送、私信、轻量画像确认、连接反馈。
- Web 负责完整资料编辑、长内容输入、深度匹配配置、管理后台入口和更复杂的信息展示。
- 两端共享 BFF/API 契约、设计系统、埋点规范和状态模型。

这里的“双端同步”指产品规划与接口设计同步，不等于所有后端域能力在同一阶段全部成熟。匹配、私信和安全闭环应按 `rules/08-ROADMAP.md` 与 `rules/09-NEXT-DEVELOPMENT-PLAN.md` 的阶段拆分渐进接入。

App 技术框架必须在 Phase 1 结束、Phase 2A 启动前完成决策。候选范围限定为 Flutter、React Native、KMP + Compose Multiplatform 或原生双栈；决策依据包括团队经验、推送/本地存储/埋点生态、性能、招聘成本和与 Web 设计系统的复用程度。框架未定前，不得把 Phase 2A 解释为完整 App 研发已经实质开工。

### 5.2 电脑客户端后置

桌面客户端不进入 MVP 主战场。它适合在正式上线运营并积累一定用户量后开发，用于长任务、常驻 Agent、浏览器协作、职业/创作/研究类深度场景。

### 5.3 端侧体验原则

- 对话是容器化入口，不把 AI 框死在固定节点工作流里。
- Skill 是成功路径的沉淀，不是静态工具列表。
- UI 层可以无限回看，推理层必须有限预算。
- 用户必须能查看、导出、删除和纠正自己的关键记忆。

## 6. 不做什么

- 不做未经授权的现实世界任意找人。
- 不为了增长牺牲隐私、反骚扰和合规。
- 不在 MVP 承诺完整音视频、直播、E2EE 全量能力。
- 不把 AI 好朋友设计成操控用户依赖的情感产品。

## 7. 多语言与区域策略

OneLink 面向全球用户，但多语言和多区域能力必须分阶段建设：

- MVP 默认先支持简体中文和英文产品框架；新增语言必须同步更新 Lumi 语气、拒绝话术、安全提示和法务文案。
- App/Web 必须预留用户语言、地区、时区、内容展示语言和通知语言字段，不把中文文案硬编码进协议。
- Lumi 多语言表现必须保持同一人格边界：温暖、可信、不过度依赖诱导，高风险场景优先安全和合规。
- 多区域扩展前必须明确数据驻留、跨境传输、删除/导出/纠正权利和本地监管要求，详见 `rules/13-GLOBAL-I18N-AND-COMPLIANCE.md`。
