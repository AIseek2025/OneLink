# Matching Safety And Governance

## 1. 匹配原则

OneLink 的匹配不是简单相似度排序，而是“理解用户意图 + 安全过滤 + 结构化召回 + 语义补充 + 反馈学习”的闭环。

## 2. 标准找人链路

1. Lumi 或前端收集用户找人意图。
2. LLM/小模型将意图结构化为任务类型和约束。
3. `safety-service` 做请求风险预审。
4. `match-service` 先走硬规则与权限过滤。
5. 结构化查询和关系图筛选候选。
6. 必要时使用向量检索补充长文本语义。
7. 排序、去重、多样性控制。
8. 返回可解释名片。
9. 曝光、点击、关注、私信、举报反馈入事件总线。

## 3. 图优先、向量补充

从 `docs/0327` 吸收“图优先、规则优先、向量补充”的思想。这里冻结的是架构原则，不冻结某篇研究稿中的具体术语、Prompt 模板或实现细节：

- 意图路由和任务选择优先使用 LLM/规则输出结构化 JSON。
- 找谁优先走关系型数据、认知图谱和明确条件。
- 向量检索用于个人简介、作品、聊天片段、长文本等细节补充。
- 不把 embedding 相似度当作唯一匹配依据。

## 4. Safety Service

`safety-service` 负责：

- 找人请求审查。
- 陌生人首条私信审查。
- 主页与画像回溯审查。
- 举报工单判定。
- 处罚建议与申诉流程。
- 拉黑与屏蔽关系。

安全必须先于增长。高风险场景宁可降级为澄清、人工审核或不推荐。

## 5. Prompt Gateway

状态：候选能力，当前冻结为“主链路前置护栏”的方向，不冻结具体部署拓扑、供应商或模型实现。

吸收 Isartor / 提示词防火墙思想，OneLink 应建设 Rust 优先的 Prompt Gateway，挂在 API Gateway、BFF 或 model-gateway 前后：

- 规则过滤：长度、格式、关键词、已知越狱模板。
- 语义过滤：轻量本地模型或嵌入比对。
- 结构校验：确保进入 Skill/Agent 的输入满足 schema。
- 缓存去重：高频重复 FAQ、帮助问题、模板型请求直接命中缓存。
- 成本保护：无价值请求不得进入昂贵模型主链路。

## 6. Lumi 劝导边界

Lumi 面对违法、骚扰、操控、诈骗、自伤或侵犯隐私需求时，必须先理解情绪，再清晰拒绝，并给出健康、合法、低伤害替代路径。

MVP 阶段劝导模板必须人工审核。AutoResearch 只能优化表达节奏，不能改变底线。

## 7. 治理指标

- 举报率。
- 误伤率。
- 申诉改判率。
- 风险召回率。
- 陌生人私信违规率。
- 推荐后有效连接率。
- 用户删除/纠正记忆处理成功率。
