# App Delivery Plan

## 1. 目标定位

App 不是 Web 的附属端，而是 OneLink 的核心产品载体之一。

后续开发必须把 App 视为：

- 用户首次建立关系的主入口
- Lumi 长期陪伴、持续对话和记忆沉淀的主载体
- 匹配、推荐、私信和安全反馈的最高频触点
- 未来国际化、多区域、通知和增长实验的主要承载端

因此，App 规划标准必须高于“能跑起来”，而要达到：

- 主链路完整
- 信息架构清晰
- 与 Rust/BFF/规则体系一致
- 可持续迭代，不因快速上线形成技术债黑洞

## 2. 技术路线

## 2.1 技术选型

App 推荐采用：

- `React Native + TypeScript` 作为主应用框架
- 共享 `repo/apps/web` 的设计 token、接口模型和埋点协议
- 通过 `BFF` 统一接入后端

App IA、页面树和状态流见 `rules/16-APP-IA-AND-STATE-FLOWS.md`。  
App/Web 共用客户端契约冻结见 `rules/17-BFF-CLIENT-CONTRACT-FREEZE.md`。

原因：

- 当前 Web 已经是 TypeScript/React 方向，前端认知和资产可复用
- App 与 Web 共享类型、接口、埋点、文案资源和设计 token 的成本更低
- Rust 仍然作为平台主语言，承担后端主链路；移动端不需要为了“统一语言”而强行自研复杂跨端 UI 栈

## 2.2 与 Rust / Go 的关系

- Rust 继续承担身份、聊天、记忆、画像、问题、匹配、安全、Model Gateway 等主服务
- Go 只在需要高并发接入层、网关配合、工具型组件时做辅助，不让 App 直接依赖 Go 私有逻辑
- App 只通过 `BFF` 和面向客户端冻结的 OpenAPI 访问系统
- 若后期需要本地高性能能力，可引入独立 `mobile-core`，以 Rust 编写并通过 FFI 提供给 App 使用，但不作为 Phase 2A 前置条件

## 3. 仓库结构规划

建议在 `repo/apps/` 下形成：

```text
repo/apps/
  web/
  admin/
  app-server/   (Rust Axum BFF service adapter, not a mobile client)
  mobile/       (React Native + TypeScript real mobile client)
```

`repo/apps/mobile/` 需要至少包含：

- `src/`：页面、组件、状态、服务
- `src/navigation/`：路由和导航
- `src/features/`：按域拆分的功能模块
- `src/shared/`：共享 UI、hooks、utils、i18n、analytics
- `src/api/`：BFF client、DTO、adapter
- `src/state/`：认证、会话、草稿、推荐、设置
- `assets/`：图标、启动图、品牌资源
- `tests/`：单测、关键流程测试
- `README.md`：本地启动、构建、测试、发布说明

## 4. App 产品边界

## 4.1 App 负责

- 启动、注册、登录、找回会话
- Lumi 对话与主要消息体验
- 基础画像确认与轻量纠错
- 找人意图快速输入
- 推荐浏览、卡片反馈、连接动作
- 私信入口、首条消息、举报/拉黑
- 消息通知、系统通知、风控通知
- 基础设置、隐私、语言、地区、通知偏好

## 4.2 Web 负责

- 长表单资料编辑
- 大屏问卷、画像深度纠错
- 匹配规则配置和复杂解释
- 运营后台、审核后台、数据视图
- 更复杂的用户数据导出/下载流

## 4.3 双端共享

- 认证状态模型
- BFF/OpenAPI 契约
- 埋点协议
- 设计 token
- i18n 资源 key
- 安全、风控、错误码口径

## 5. App 信息架构

建议 App 第一版使用五个一级导航：

1. `首页 / Lumi`
2. `找人`
3. `推荐`
4. `消息`
5. `我的`

对应含义：

- `首页 / Lumi`：主欢迎面、会话列表、当前对话、系统引导
- `找人`：快速表达找人目标、补充条件、查看请求状态
- `推荐`：候选卡、解释、反馈、连接动作
- `消息`：系统消息、私信、审核反馈、举报进展
- `我的`：个人资料、语言、地区、通知、隐私、帮助

## 6. Phase 分解

## 6.1 App Phase A0：工程地基

目标：建立正式 App 工程，而不是占位目录。

交付物：

- `repo/apps/mobile/` 正式目录
- 路由框架
- 主题系统
- API client
- 登录态存储
- 环境配置
- 埋点 SDK 封装
- i18n 资源框架
- CI 中的 App lint/test/build 基线

验收：

- 新成员可按 README 跑起 App
- 能连接 mock BFF 或 dev BFF
- 基础导航、空状态、错误状态可见

## 6.2 App Phase A1：核心主旅程

目标：完成“注册 -> 登录 -> Lumi 聊天 -> 画像确认”的核心链路。

页面范围：

- 启动页
- 登录页
- 注册页
- 会话列表页
- 聊天页
- 画像确认卡
- 基础个人页

能力范围：

- token/session 持久化
- 首登引导
- 聊天首包加载
- 画像确认卡展示、接受、稍后处理
- 错误提示、网络异常、重试

验收：

- 内部用户可在真机完成首轮注册登录和首次聊天
- 聊天/画像埋点正确上报
- crash、白屏、无状态恢复问题可控

## 6.3 App Phase A2：找人与推荐

目标：把 App 从“对话工具”升级为“连接产品”。

页面范围：

- 找人输入页
- 条件编辑页
- 推荐列表页
- 推荐卡详情页
- 反馈页

能力范围：

- 提交找人请求
- 查看请求状态
- 推荐结果列表与解释
- 喜欢/跳过/稍后看
- 失败空态和澄清路径

验收：

- 真机演示可完成“找人 -> 收到推荐 -> 反馈”
- 推荐卡片不再是纯占位

## 6.4 App Phase A3：私信与安全闭环

目标：把推荐连接升级为受保护的真实交互。

页面范围：

- 首条私信页
- 对话详情页
- 举报页
- 拉黑页
- 安全提示与申诉状态页

能力范围：

- 首条私信风控审查
- 风险提示
- 举报、拉黑、撤回
- 审核反馈展示

验收：

- 首条私信经过 `safety-service`
- 举报/拉黑结果进入后台
- 用户可见的安全提示文案完整且可审核

## 6.5 App Phase A4：生产强化

目标：让 App 达到可持续运营和多区域扩展的标准。

范围：

- push 通知
- 离线恢复
- 资源化文案
- 中文/英文切换
- 地区与通知语言设置
- 用户数据权利入口
- 发布流水线
- 崩溃监控

验收：

- 双语切换有效
- 关键设置落库并影响 BFF 请求
- 版本发布流程稳定

## 7. 详细模块规划

## 7.1 认证模块

必须包含：

- 邮箱/手机号登录策略的明确决策
- 注册与登录错误码标准化
- 会话过期处理
- 多设备策略说明
- 退出登录和撤销能力

验收：

- 与 `identity-service` 错误码对齐
- App 不自己拼业务错误文案

## 7.2 Lumi 聊天模块

必须包含：

- 会话列表
- 消息流
- 首包加载
- typing / loading / retry 状态
- 内容安全反馈与系统提示

验收：

- App 聊天行为与 Web 聊天契约一致
- 会话恢复、失败重试、超时提示可用

## 7.3 画像确认与纠错模块

必须包含：

- 画像确认卡
- 用户接受 / 拒绝 / 稍后
- 轻量事实纠错
- 跳转到 Web 深度编辑的桥接说明

验收：

- 画像确认事件进入 profile/context 链路
- App 只做轻量确认，不承载过长复杂编辑

## 7.4 找人与推荐模块

必须包含：

- 简洁表达入口
- 条件补充
- 请求处理中状态
- 推荐卡片
- 推荐解释
- 反馈按钮

验收：

- 与 `match-service`、`bff` 契约一致
- “后续版本上线”类占位从 App 主链路中移除

## 7.5 私信与安全模块

必须包含：

- 首条消息限制
- 风险提示
- 举报与拉黑
- 申诉状态可见

验收：

- 与 `dm-service`、`safety-service`、后台审核链一致

## 7.6 我的 / 设置模块

必须包含：

- 个人基本信息
- 语言
- 地区
- 时区
- 通知语言
- 通知开关
- 隐私与帮助
- 数据权利入口

验收：

- 配置项有明确落库路径
- 不只是前端展示字段

## 8. 设计与体验要求

App 必须遵守：

- 首页和聊天优先单手操作
- 找人和推荐尽量短流程
- 高风险操作必须二次确认
- 空状态、加载态、失败态、审核中态都必须成体系
- Lumi 语气、按钮文案、安全提示在中英文中保持一致人格边界

设计系统要求：

- 共享 `tokens`
- 颜色、间距、排版、圆角、阴影一致
- 卡片、底部弹层、输入框、消息气泡统一

## 9. App 与 BFF / 后端协作清单

App 开发开始前必须冻结：

- 登录/注册/会话刷新契约
- 聊天消息发送/接收契约
- 画像确认和纠错契约
- 找人请求提交契约
- 推荐列表与反馈契约
- 首条私信与风险提示契约
- 设置与 i18n 配置契约

App 不直接拼接内部服务，不绕过 BFF。

详细冻结范围见 `rules/17-BFF-CLIENT-CONTRACT-FREEZE.md`。

## 10. App 质量门禁

必须建立：

- lint
- unit test
- 关键主旅程集成测试
- API mock 回归
- 版本发布说明
- 崩溃与性能监控

上线前必须确认：

- 登录、聊天、找人、推荐、消息、设置六条主链路至少各有一条可重复测试
- 弱网、超时、接口失败、鉴权过期有明确处理
- 不出现明显白屏、导航死路、状态错乱

## 11. App 与全球化 / 合规的前置要求

App 不允许后补这些底层问题：

- 用户可见文案硬编码
- 安全拒绝文案未经审核
- 语言、地区、通知语言混在一个字段里
- 用户数据导出/删除/纠错完全没有入口

Phase A1 开始就必须预埋：

- i18n key
- locale store
- 合规入口占位
- 风险文案审核流程

## 12. 团队配置建议

App 需要最低配置：

- 移动端主程 1 人
- 移动端工程 / 交付支持 1 人
- 设计 1 人
- BFF / Rust 后端对接 1 人
- QA 1 人

若资源不足，优先保证：

- A0 工程地基
- A1 核心主旅程
- A2 找人和推荐

不要为了赶进度提前拆散质量门禁。

## 13. 交付节奏建议

建议按四个开发波次执行：

1. `Wave 1`：A0 工程地基
2. `Wave 2`：A1 核心主旅程
3. `Wave 3`：A2 找人与推荐
4. `Wave 4`：A3 私信与安全闭环

A4 生产强化与全球化配套可和 `rules/14-DELIVERY-CLOSURE-PLAN.md` 的 `Track D/E` 并行推进。

对应执行手册见：

- `rules/19-CAPACITY-SLO-EXECUTION-HANDBOOK.md`
- `rules/20-GLOBAL-I18N-COMPLIANCE-EXECUTION-HANDBOOK.md`

## 14. 最终目标

App 完成后，应达到：

- 它是 OneLink 用户最自然、最稳定的主入口
- 它不是 Web 的阉割镜像，而是承担核心主链路的第一产品端
- 它与 Rust 后端、BFF、规则、容量、安全、全球化规划完全对齐
- 它具备后续推向真实用户和多区域扩展的工程基础
