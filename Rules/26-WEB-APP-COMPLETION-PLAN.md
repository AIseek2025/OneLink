# Web + App Completion Plan

## 1. 目标

本文件用于把 OneLink 从“已有阶段性交付基线”推进到“Web 与 App 双端都开发完毕并达到上线条件”的执行计划。

这里的“开发完毕”不是指：

- 只有文档、截图或占位页
- 只有后端接口，没有真实客户端
- 只有能演示的 happy path，没有构建、测试和上线门禁

这里的“开发完毕”必须同时满足：

- Web 端主链路、后台与设置能力完整可用
- App 端存在真实移动客户端工程，并完成主链路与生产强化
- 双端通过统一 BFF 契约、埋点、设计 token、i18n 资源与合规口径协同
- 代码、契约、测试、CI、运行验证和发布说明全部齐备

## 2. 当前真实状态

当前代码状态不能简单沿用对外交付报告中的“已完成”表述，而应以仓库事实为准。

### 2.1 Web 端现状

- `repo/apps/web/` 已是正式 React + TypeScript + Vite 工程，具备 `build / test / typecheck` 脚本。
- Web 已接入登录、首页、聊天、画像、问卷、找人、推荐、私信、安全等主链路接口。
- 但 `FindPage` 仍显示“匹配功能将在后续版本上线”。
- `AdminPage` 仍显示“管理后台功能将在后续版本上线”。
- `client.ts` 已统一收口到 `/api/v1/bff/*`，不再混用 `/api/v1/app/*`。

结论：Web 已部分成型，但还没有达到“完整产品前台 + 最小后台 + 正式上线基线”。

### 2.2 App 端现状

- 当前 `repo/apps/app/` 是 Rust Axum 启动器 crate（`onelink-app`），依赖 `repo/apps/app-server/` 库 crate（`onelink-app-server`）；两者均为服务端工程，不是真实移动端客户端。
- 仓库内已建立 `repo/apps/mobile/`（React Native + TypeScript）作为真实移动端客户端骨架。
- 这意味着“App 接入层”已经存在，但“App 产品端”实际上尚未落地。

结论：App 端当前不能判定为已开发成型，必须新增真实移动端工程并交付。

### 2.3 后端与契约现状

- BFF、identity、profile、match、dm、safety 等服务已有较完整实现。
- OpenAPI、contract test、Rust workspace、CI 与审计证据已具备较好的工程基础。
- 仍有尾项需要收口，例如占位路由残留、双端 API 边界不纯、上线前的完整烟测与持久化实证不足。

结论：后端与契约层可作为双端完工的基础，但不能替代双端自身的完工验收。

## 3. 目标完成态定义

### 3.1 Web 完成态

Web 端完成后必须达到：

- 用户前台主链路完整：注册、登录、聊天、画像确认/编辑、找人、推荐、首条私信、安全反馈、设置、合规入口
- 运营后台最小可用：审核队列、举报处理、申诉查看、基础指标看板
- 所有“后续版本上线”“pending 占位”类前台文案从主链路中移除
- 所有用户可见文案完成资源化，不依赖硬编码
- 所有前端 API 统一通过冻结后的客户端接入层，不再混用无边界路径

### 3.2 App 完成态

App 端完成后必须达到：

- 仓库内存在真实移动端工程，而不是服务适配层冒充客户端
- 支持 iOS/Android 开发构建、调试、测试、打包与发布说明
- 主链路完整：注册、登录、Lumi 聊天、画像确认、找人、推荐、私信、安全、设置、合规入口
- 支持中英文切换、地区/时区/通知语言设置、弱网与异常恢复
- 支持埋点、崩溃监控、版本发布和回滚说明

### 3.3 双端共同完成态

双端共同完成后必须达到：

- 共用 BFF 契约、错误码口径、埋点协议、i18n key、设计 token
- 关键链路均有自动化测试、联调脚本和 smoke 证据
- 发布前必须通过 build、lint、test、contract、smoke、health check
- 对内对外文档与仓库事实一致，不再出现“代码未完成但报告声称完成”

## 4. 执行原则

- 先收口工程形态，再补功能，最后做生产强化；不允许跳过基础工程直接堆页面。
- App 必须是独立真实客户端，不允许继续把服务适配层视为 App 交付物。
- Web 与 App 不能各自私接后端，必须通过统一冻结后的 BFF 契约。
- 功能完成不等于交付完成；所有阶段都必须带构建、测试、文档和运行证据。
- 所有可上线判断都以代码、测试、构建和健康验证为准，不以口头结论为准。

## 5. 总体路径

本计划按 5 个 Wave 执行，建议以 8-10 周为目标窗口推进。若资源不足，可以拉长周期，但不得跳过 Wave 0 或 Wave 4。

### Wave 0：工程重整与边界冻结

目标：把当前“App 目录误指向服务”“Web/API 分层不纯”“文档口径过乐观”的问题先收口。

必须完成：

- 明确真实移动端技术栈，沿用 `rules/15-APP-DELIVERY-PLAN.md` 的 `React Native + TypeScript` 方案
- 确保 `repo/apps/app/`（Axum 启动器）与 `repo/apps/app-server/`（路由库）的命名不与真实 App 客户端 `repo/apps/mobile/` 混淆
- 建立正式移动端目录结构，并补齐 README、脚本、环境变量规范
- 收口 Web/API 接入层，冻结 `/api/v1/bff/*` 作为客户端主入口
- 盘点并清理 BFF 中的 placeholder 与阶段性残留路由
- 校正 `rules/`、README、交付口径与代码事实的偏差

验收标准：

- 仓库中存在清晰的 `真实 App 客户端` 与 `服务适配层` 边界
- Web 与 App 客户端 API 入口统一
- 工程结构、命名、README、契约说明一致

### Wave 1：Web 产品闭环补完

目标：让 Web 从“能演示”升级为“完整前台 + 最小后台”。

必须完成：

- 完成 `Find` 真实结果态、澄清态、失败态、空态与解释链路
- 完成 `Recommendations` 详情、反馈、连接动作与状态回流
- 完成 `DM` 线程列表、首条私信、风控提示、失败重试
- 完成 `Safety` 的举报、拉黑、申诉状态可见
- 完成 `Settings` 的语言、地区、时区、通知语言、隐私与帮助入口
- 完成 `Compliance` 的查看、导出、更正、删除入口与结果态
- 完成 `Admin` 最小可运营后台：举报队列、申诉查看、基础指标、审核状态
- 移除前台主链路中的所有占位文案和假数据卡片

验收标准：

- Web 用户可完成注册到私信的完整链路
- Web 管理员可完成基础审核与查询链路
- `npm run build`、`npm run test`、`npm run typecheck` 全部通过
- Web smoke 覆盖登录、找人、推荐、私信、安全、设置、合规、后台

### Wave 2：真实 App 客户端落地

目标：在仓库内交付真正的移动端客户端，不再以服务端工程代替 App。

必须完成：

- 建立 React Native 工程骨架、导航、状态管理、主题、埋点、i18n、配置体系
- 打通登录态存储、BFF client、错误处理、空态/失败态/加载态
- 完成 App 一级导航：`首页/Lumi`、`找人`、`推荐`、`消息`、`我的`
- 完成注册、登录、聊天、画像确认、找人输入、推荐列表、设置基础页
- 建立真机调试、模拟器运行、测试命令、构建命令和发布文档

验收标准：

- 新成员可根据 README 在本地跑起 App
- iOS/Android 至少各完成一轮开发构建验证
- App 主链路可在真机或模拟器上完整演示
- App lint、unit test、关键流程集成测试可重复执行

### Wave 3：双端统一闭环

目标：让 Web 与 App 都完成相同的核心用户闭环，而不是一端完整、一端半成品。

必须完成：

- 找人请求、推荐结果、反馈、首条私信、安全审查在双端表现一致
- 双端共享埋点协议与事件命名，不再出现口径不一致
- 双端共享错误码、状态机和空态/失败态策略
- 双端共享 i18n key、设计 token 和合规入口结构
- 后台审核处理结果能够反向影响双端用户界面

验收标准：

- Web 与 App 都能完成 `注册 -> 登录 -> 找人 -> 推荐 -> 首条私信 -> 举报/拉黑`
- 同一条后端契约在双端消费方式一致
- 双端截图、演示、测试结果与产品说明一致

### Wave 4：生产强化与上线门禁

目标：把“功能做完”推进到“可以上线”。

必须完成：

- 双端接入崩溃监控、版本信息、构建产物管理和回滚说明
- 完成双端中英文资源化和地区/时区/通知语言设置落库
- 完成双端合规入口与数据权利接口联通
- 补齐弱网、超时、鉴权过期、接口失败、服务降级、重试策略
- 建立双端 smoke、BFF health check、核心服务 health check、上线检查清单
- 完成一次正式预发演练：build、部署、烟测、API 健康检查、回滚演练

验收标准：

- Web build 通过且可部署预发环境
- App 开发构建与发布构建都通过
- API 健康检查、前端 smoke、关键路径回归全部通过
- 所有上线文档、脚本、报告和 evidence 可复核

## 6. 详细工作流

### 6.1 Web 工作流

Web 端按以下顺序推进：

1. 收口 API client 与页面状态机
2. 补齐 Find / Recommendation / DM / Safety 主链路
3. 补齐 Settings / Compliance / Locale / Region
4. 实现 Admin 最小后台
5. 统一埋点、i18n、错误处理、组件规范
6. 完成测试、烟测和预发验证

### 6.2 App 工作流

App 端按以下顺序推进：

1. 清理目录冲突并建立真实 App 工程
2. 接入统一 BFF client、认证态与导航
3. 完成聊天、画像确认、找人、推荐
4. 完成私信、安全、设置、合规
5. 接入 i18n、通知、监控、发布与回滚
6. 完成真机联调、测试和发布验证

### 6.3 BFF 与后端协同工作流

双端开发期间，后端必须并行完成：

- 客户端契约冻结与破坏性变更管控
- 推荐、私信、安全、合规接口的真实化和稳定性补强
- 错误码、状态枚举、埋点字段、DTO 的统一
- 集成测试、contract test、persistence smoke 与 health check 增强

## 7. Definition of Done

### 7.1 Web DoD

- 页面与接口都是真实实现，不是占位文案或假数据
- 所有主链路页面均有 loading、empty、error、retry 状态
- 所有用户可见文案来自资源文件
- `build / test / typecheck` 通过
- 有 smoke 脚本和预发验证记录

### 7.2 App DoD

- 存在真实移动端工程与可运行入口
- iOS/Android 都有构建与运行说明
- 主链路和设置/合规入口完成
- 有 lint、unit test、集成测试和发布说明
- 有真机或模拟器运行证据

### 7.3 系统 DoD

- Web、App、BFF、后端契约一致
- 双端关键埋点一致，事件名与 schema 一致
- 双端 build 全通过，服务 health 全通过
- smoke、回归、健康检查通过后才允许声明“可上线”

- 登录、聊天、找人、推荐、私信、安全、设置、合规的全链路烟测

## 9. 角色与分工建议

| 模块 | 主 owner | 协作 owner | 主要产物 |
|------|----------|-------------|----------|
| Web 闭环补完 | Web 负责人 | BFF、设计、QA | 页面、组件、E2E、后台 |
| App 客户端落地 | 移动端负责人 | BFF、设计、QA | RN 工程、导航、状态、构建发布 |
| BFF/契约收口 | Rust 后端 | Web、移动端 | OpenAPI、DTO、错误码、集成测试 |
| 推荐/私信/安全 | Rust 后端 | Web、移动端、运营 | match/dm/safety 闭环 |
| i18n/合规/区域 | 产品 + 后端 | Web、移动端 | 资源文件、数据权利、区域策略 |
| 上线门禁 | 平台 / QA | 全体 | smoke、health、发布/回滚文档 |

## 10. 风险与前置约束

- `repo/apps/app/`（Axum 启动器，`onelink-app`）与 `repo/apps/app-server/`（路由库，`onelink-app-server`）均为服务端工程；真实 App 客户端位于 `repo/apps/mobile/`。
- 如果先做 App UI、后补契约冻结，双端会快速分叉。
- 如果不先移除 Web 占位页，后续测试和对外交付口径会继续失真。
- 如果不把 i18n、地区、合规入口提前预埋，后期返工成本会显著上升。
- 如果没有预发 build + API 健康检查，不允许宣布上线完成。

## 12. 结论

OneLink 下一阶段的正确目标，不是继续沿用“阶段性交付已完成”的静态表述，而是把以下事情真正做完：

- 把 Web 从部分成型补到完整可上线
- 把 App 从“尚无真实客户端”补到正式移动端产品
- 把双端能力、契约、埋点、i18n、合规和上线门禁统一收口

只有在双端代码、测试、构建、烟测、健康检查和发布证据都完整的前提下，OneLink 才能被认定为“Web 与 App 都开发完毕并达到上线条件”。
