# Phase 2B Closure Checklist

## 1. 目的

本文件用于把 `Phase 2B / P2` 从“已有骨架”推进到“真实闭环已完成”。

OneLink 只有在以下链路全部成立后，才能宣称 Phase 2B 完成：

- 找人请求可真实提交
- 推荐结果可真实消费
- 首条私信可真实发起
- 安全审查可真实拦截
- 举报 / 拉黑 / 反馈可真实回流
- 管理后台可真实查看关键队列

## 2. 当前缺口定义

Phase 2B 当前不通过的典型信号包括：

- 页面仍写“后续版本上线”
- 服务只有路由或内存态样例，没有真实接线
- BFF 返回 mock/placeholder 而不是 owner service 结果
- 推荐、私信、安全、后台之间没有闭环回流

## 3. 收口范围

必须覆盖以下对象：

- `Find`
- `Recommend`
- `DM`
- `Safety`
- `Admin`
- `BFF`
- `match-service`
- `dm-service`
- `safety-service`

## 4. Find 收口清单

必须完成：

- 用户可提交真实找人请求
- 请求进入澄清 / 审核 / 召回流程
- 用户能查看请求状态
- 空结果和澄清路径不是占位文案

验收：

- App/Web 都能提交请求
- 请求状态能从 BFF 读到
- 至少 1 条真实候选路径可演示

## 5. Recommend 收口清单

必须完成：

- 推荐列表从真实 owner service 返回
- 推荐详情有解释摘要
- 喜欢 / 跳过 / 稍后看写回反馈事件
- 无推荐结果时给出真实可恢复路径

验收：

- 推荐卡片不再是静态样例
- 反馈会影响后续状态或事件流

## 6. DM 收口清单

必须完成：

- 首条私信创建和发送
- 审核中状态
- 审核通过后进入真实线程
- 审核失败后返回用户可理解说明

验收：

- App/Web 都可触发最小私信链路
- `dm-service`、BFF、前端状态一致

## 7. Safety 收口清单

必须完成：

- 首条私信安全审查
- 举报提交
- 拉黑执行
- 申诉状态查询
- 风险解释文案接入审核版内容

验收：

- 审查可阻断发送
- 举报 / 拉黑会写入后台可见队列
- 高风险提示不是硬编码临时文案

## 8. Admin 收口清单

必须完成：

- 举报队列
- 风控结果查看
- 申诉队列
- 最小审核动作

验收：

- Admin 不再是占位页
- 至少一条“举报 -> 审核 -> 结果可见”的链路可演示

## 9. 后端接线要求

后端必须满足：

- `match-service` 具备真实候选返回
- `dm-service` 具备真实线程和审核状态
- `safety-service` 具备真实决策和动作写入
- `bff` 聚合响应与 App/Web 状态流一致

禁止：

- 只改前端页面，不接真实服务
- 只改 README，把占位字样删掉却没有真实链路

## 10. 测试与验收

Phase 2B 必须补齐：

- contract test
- BFF smoke
- App/Web 主旅程测试
- 报告 / 拉黑 / 申诉至少一条回归

建议最小演示旅程：

1. 用户注册并完成基础画像
2. 用户提交找人请求
3. 用户收到推荐
4. 用户查看解释并发起连接
5. 用户发送首条私信
6. 安全审查通过或拦截
7. 用户举报或拉黑
8. 后台看到记录并处理

## 11. 完成判定

以下任一情况仍存在时，Phase 2B 不能判定完成：

- 页面保留“后续版本上线”
- Admin 仍是占位
- 推荐只是 mock
- 首条私信绕过 safety
- 举报 / 拉黑没有后台回流
