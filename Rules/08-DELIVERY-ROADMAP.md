# OneLink Delivery Roadmap

## 1. 路线图原则
- 先验证连接闭环，再扩展规模
- 先把模型接口和数据骨架搭好，再逐步替换底层模型
- 每个阶段只解决该阶段最关键的瓶颈，不提前背上过重的复杂度

## 2. Phase 0：架构冻结与原型验证

### 2.1 目标
- 冻结产品边界、主技术栈、模型网关和数据骨架
- 做出可演示原型

### 2.2 交付物
- 本次生成的 9 份规范文档
- 交互原型
- 领域模型和服务边界草图
- MVP 指标定义

### 2.3 成功标准
- 团队对核心边界无重大分歧
- 没有“技术栈反复横跳”
- 可以开始进入工程实施

## 3. Phase 1：MVP

### 3.1 目标
- 验证“聊天 -> 记忆 -> 画像 -> 找人 -> 推荐 -> 私信”的最小闭环

### 3.2 范围
- 账户系统
- AI 聊天入口
- 基础问卷系统
- 基础必填题包
- 基础主页
- 基础事实抽取
- 找人请求
- 风险审查
- 名片推荐
- 关注
- 陌生人单条私信
- 投诉入口

### 3.3 技术策略
- 单区域部署
- 外部大模型为主
- MVP 核心后端统一 `Rust`
- 从第一天就落地 `BFF`
- 从第一天就落地 `context-service`（记忆计算层）
- 从第一天拆分 `ai-chat-service` 与 `dm-service`
- PostgreSQL + Redis + Kafka + 对象存储
- 向量检索用轻量方案

### 3.4 核心指标
- 注册转化率
- 找人请求完成率
- 推荐点击率
- 私信发起率
- 私信回复率
- 举报率

## 4. Phase 2：数据飞轮与产品增强

### 4.1 目标
- 让画像、问题、推荐和治理形成反馈闭环

### 4.2 新能力
- 动态追问
- 更强的画像摘要
- 更强的风控分类
- 更完整的举报和处罚链路
- 推荐精排优化

### 4.3 技术策略
- 引入更独立的向量服务
- 引入基础特征服务和批处理任务
- 开始自托管小模型

### 4.4 切换门槛
- 核心闭环指标达到预期
- 样本量足够支撑小模型训练
- 运营已经积累真实投诉和误判样本

## 5. Phase 3：规模化准备

### 5.1 目标
- 支撑百万到千万级用户增长
- 降低高频模型成本
- 提升稳定性和可观测性

### 5.2 新能力
- 服务拆分更清晰
- 独立排名服务
- 扩大 `question-service` 的动态题库和试投放能力
- 内部研究集群
- 统一审核控制台
- 更完整的缓存和降级策略

### 5.3 技术策略
- 双区域容灾
- 自托管领域模型承接高频能力
- 统一实验平台和灰度发布

### 5.4 切换门槛
- 外部模型成本明显上升
- 单区域可用性不足
- 事件与消息吞吐逼近当前架构边界

## 6. Phase 4：全球化与多区域治理

### 6.1 目标
- 支撑跨区域服务
- 满足数据驻留和主要地区合规要求
- 提升多语言和跨时区匹配质量

### 6.2 新能力
- 区域化数据管理
- 多区域路由
- 区域模型策略
- 更完整的审计和导出删除能力

### 6.3 技术策略
- 多区域控制平面
- 数据按区域存储
- 训练与推理按区域治理

### 6.4 切换门槛
- 海外用户规模显著增长
- 合规要求开始成为增长瓶颈
- 跨区域延迟影响核心体验

## 7. Phase 5：自有模型组合与组织级自动化

### 7.1 目标
- 自有模型承接多数高频场景
- 自动研究与 Agent 治理成熟

### 7.2 新能力
- 自有聊天中模型
- 自有安全模型
- 自有抽取和推荐辅助模型
- 内部 Agent 控制面
- 实验自动化升级

### 7.3 成功标准
- 高频能力大部分不再依赖外部 API
- 成本结构更健康
- 模型迭代周期明显缩短

## 8. 团队配置建议

### 8.1 Phase 1
- 产品负责人
- 前端工程师
- Rust 工程师（主力）
- Go 工程师（辅助工具链，可由 Rust 工程师兼任）
- Python/AI 工程师
- 测试与质量工程师
- 设计师

### 8.2 Phase 2 到 Phase 3
- 数据工程师
- 安全工程师
- 运维/平台工程师
- 审核运营负责人
- 推荐和风控算法工程师

### 8.3 Phase 4 以后
- 区域合规负责人
- SRE 团队
- 训练平台团队
- 多语言运营和安全团队

## 9. 里程碑与门槛

### 9.1 产品门槛
- 有效连接率达到内部预期
- 举报率在可控范围
- 申诉改判率不失控

### 9.2 技术门槛
- 模型网关稳定
- 事件骨干稳定
- 主链路延迟稳定
- 监控和回滚链路成熟

### 9.3 组织门槛
- 文档先行
- 发布门禁稳定运行
- 事故复盘机制生效

## 10. 不要过早做的事
- 不要一开始就上过多微服务
- 不要一开始就承诺完整音视频、直播和 E2EE 全量能力
- 不要在没有高质量样本前就投入大规模自研训练
- 不要把全球化当作第一阶段任务，而应先把骨架设计对

## 11. 每阶段必须保留的底座
- 模型网关
- 事件骨干
- 审计日志
- 回滚能力
- 指标体系
- 投诉和处罚闭环

## 12. 最终判断标准
OneLink 的路线图不是“功能越多越好”，而是每个阶段都要持续回答：
- 是否更容易找到合适的人？
- 是否更安全？
- 是否更可扩展？
- 是否更接近自有模型和全球规模的长期目标？
