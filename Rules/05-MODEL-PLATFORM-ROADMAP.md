# OneLink Model Platform Roadmap

## 1. 文档目标
- 定义 OneLink 如何在 MVP 阶段使用外部大模型
- 定义如何从第一天就为自研和自托管模型留出稳定接口
- 定义数据、训练、推理、评估和发布的统一平台边界

## 2. 核心结论

### 2.1 总策略
- `短期`：外部顶级模型 API 承接复杂通用能力
- `中期`：自建领域小模型接管高频、结构化、低延迟场景
- `长期`：形成 OneLink 自有模型组合，外部模型仅处理少量高复杂度或兜底场景

### 2.2 绝对前提
- 所有模型能力必须通过 `Model Gateway`
- 所有 Prompt、路由、缓存、审计、成本、降级统一管理
- 业务系统永远只依赖能力接口，不依赖某一家模型供应商
- 长期记忆与上下文组装统一由 `context-service` 提供，业务服务不能绕过它直接拼接长期上下文

## 3. 模型能力分层

### 3.1 Reflex Layer
- 目标：极低延迟、极低成本、极高可控性
- 典型任务：
  - 意图识别
  - 文本分类
  - 敏感词和规则增强
  - 轻量风险初筛
  - 向量生成
- 来源：
  - 传统模型
  - 小型 Transformer
  - 高性能规则引擎

### 3.2 Domain Layer
- 目标：承接 OneLink 特有任务
- 典型任务：
  - 画像事实抽取
  - 问题生成与改写
  - 推荐理由生成
  - 投诉摘要
  - 安全复核
- 来源：
  - 开源模型微调
  - 自托管推理集群

### 3.3 Frontier Layer
- 目标：处理复杂推理、开放式对话、多语言复杂理解
- 典型任务：
  - 深度陪伴对话
  - 长上下文综合判断
  - 高复杂度澄清问题生成
  - 困难案例安全分析
- 来源：
  - 外部顶级模型
  - 后期自有大型模型

## 4. Model Gateway

### 4.1 网关职责
- 能力路由
- Provider 抽象
- Prompt 模板管理
- 响应规范化
- 缓存
- 降级
- 审计
- 成本计量
- 区域策略

### 4.1.1 与 `context-service` 的边界
- `context-service` 决定：
  - 哪些记忆被选入上下文
  - working memory / persistent memory 的组织方式
  - token budget 的前置控制
- `model-gateway` 决定：
  - 用哪个模型
  - 用哪个 Prompt 版本
  - 是否缓存
  - 是否降级
- 结论：
  - `context-service` 负责“喂什么”
  - `model-gateway` 负责“怎么调”

### 4.2 统一能力接口
- `chat.respond`
- `profile.extract_facts`
- `profile.summarize`
- `question.generate`
- `question.review`
- `safety.classify_request`
- `safety.review_message`
- `match.explain`
- `embedding.encode`

### 4.3 标准输出
- 所有结构化任务输出 JSON schema
- 所有自由文本任务输出都必须附带：
  - model_id
  - prompt_version
  - latency_ms
  - cost_estimate
  - safety_flags

## 5. Prompt 管理

### 5.1 Prompt 生命周期
1. 业务定义能力目标
2. 平台定义 schema 和约束
3. 产出 prompt 模板
4. 灰度验证
5. 版本冻结
6. 指标跟踪
7. 定期淘汰和重写

### 5.2 Prompt 结构规范
- `system`
  - 角色、边界、风格、安全约束
- `context`
  - 来自 `context-service` 的：
    - working memory summary
    - persistent memory artifacts
    - 画像摘要
    - 业务配置
- `instruction`
  - 当前任务目标
- `output_contract`
  - JSON schema 或明确格式

### 5.3 禁止事项
- 禁止直接把原始长对话无脑拼进 Prompt
- 禁止业务服务自己在本地拼长期记忆后再调模型
- 禁止在 Prompt 中包含无授权的敏感信息
- 禁止每个服务自己私下维护一套 Prompt 模板

## 6. 外部模型接入策略

### 6.1 MVP 推荐
- 复杂对话：外部领先通用模型
- 复杂推理：可与对话主模型分离，采用独立推理模型
- embedding：稳定、成本可控的嵌入服务
- 安全复核：高一致性模型 + 规则
- 问题生成：高质量文本模型
- 建议口径：
  - MVP 可采用“对话主模型 + 推理辅助模型”的双模型路由
  - 由 `model-gateway` 按能力配置具体供应商
  - 不在业务代码和规则文档中写死具体品牌名

### 6.2 接入原则
- 至少双供应商可切换
- 关键能力必须有降级策略
- 所有调用成本按能力、场景、用户等级可见
- 区域合规要求高的任务优先使用可控部署或本地化模型
- 上游调用方优先以 `context-service` 产出的结构化上下文包为输入，而不是直接上传原始长期对话

### 6.3 降级策略
- 首选模型失败时切备选模型
- 高复杂度能力不可用时降级到简化版输出
- 风险场景永远不能“静默放行”，只能拒绝、延迟或人工复核

## 7. 自研模型路线

### 7.1 阶段一：能力抽象先行
- 在没有自研模型之前，先定义好能力接口和评估体系
- 目标：未来替换底层模型时，业务代码无需重写

### 7.2 阶段二：领域小模型
- 优先自研或自托管的能力：
  - 风险分类
  - 画像抽取
  - 问题审核
  - 推荐理由生成
- 这些能力高频、结构化、容易评估、回报率最高

### 7.3 阶段三：领域中模型
- 在具备足够脱敏高质量数据后，建设中等规模领域模型承接：
  - 日常聊天
  - 问题生成
  - 画像摘要
  - 多语言匹配辅助

### 7.4 阶段四：自有大型模型组合
- 不是一个模型做所有事情，而是形成组合：
  - Chat Model
  - Safety Model
  - Extraction Model
  - Ranking Assistant
  - Embedding Model

## 8. 开源底座建议

### 8.1 主选
- `Qwen` 系列
  - 中文和多语言能力平衡好
  - 适合 OneLink 的全球化和中文起步路线

### 8.2 备选
- `Llama` 系列
  - 社区成熟
  - 生态工具丰富

### 8.3 选型标准
- 商业许可
- 多语言表现
- 长上下文
- 推理成本
- 微调友好性
- 推理部署成熟度

## 9. 训练平台

### 9.1 数据来源
- 用户授权对话
- 问卷回答
- 推荐反馈
- 投诉与处罚结果
- 人工审核结果

### 9.2 数据分层
- `Raw`
  - 原始事件
- `Curated`
  - 清洗、脱敏、结构化后的训练样本
- `Gold`
  - 人工和规则共同校验的高质量样本

### 9.3 训练任务
- 监督微调
- 排序学习
- 对比学习
- 分类器训练
- 奖励模型和策略评估

## 10. 评估与发布

### 10.1 统一评估框架
- 离线评估
  - 准确率
  - 召回率
  - F1
  - NDCG
  - 延迟
  - 成本
- 在线评估
  - 点击率
  - 回复率
  - 举报率
  - 留存
  - 误伤率

### 10.2 发布门禁
- 没有离线基准对比，不允许上线
- 没有回归安全测试，不允许上线
- 没有成本评估，不允许扩大流量
- 没有回滚方案，不允许正式切换

## 11. 推理基础设施

### 11.1 MVP
- 主要依赖外部 API
- 自托管只承接小模型和分类器

### 11.2 成长期
- 部署领域模型推理集群
- 引入批处理和异步推理队列
- 引入模型缓存和 Prompt Cache
- 与 `context-service` 联动做 memory-aware routing

### 11.3 规模化
- 推理集群按区域部署
- 按能力进行 GPU/CPU 分层
- 高频轻任务优先小模型
- 复杂低频任务才进入大模型

## 12. 为什么这条路线不会推倒重来
- 业务层从第一天就只调用能力接口
- 数据层从第一天就保留训练和评估所需事件
- Prompt、模型、排序、缓存、审计都统一纳入网关和平台治理
- 自研模型不是替换产品架构，而是替换 `Model Gateway` 背后的实现

## 13. 决策门槛

### 13.1 何时启动领域小模型
- 外部模型成本开始明显侵蚀毛利
- 画像抽取、风控分类已经有足够样本
- 对延迟和合规要求高于外部 API 可满足范围

### 13.2 何时启动中模型或大型模型
- 日常高频场景已被小模型覆盖但仍有明显成本压力
- 已经形成稳定的脱敏训练数据飞轮
- 团队具备推理、训练、评估和安全治理能力

## 14. 禁止事项
- 禁止业务服务直接绑定某一家模型 SDK
- 禁止把训练数据和生产密钥混放
- 禁止没有审计和脱敏就把用户原始数据喂给训练流程
- 禁止因为追求回复质量而跳过安全复核
- 禁止把 `context-service` 与 `model-gateway` 混成一个超级服务
